🔴 OUTAGE — 2021 / Facebook
6 שעות בלי Facebook, Instagram ו-WhatsApp בגלל שורת קונפיג אחת
ב-4 באוקטובר 2021, מהנדס ב-Facebook ביצע שינוי ב-BGP (Border Gateway Protocol) — פרוטוקול שמספר לנתבים ברחבי האינטרנט איך להגיע לשרתי Facebook. הפקודה ששלח הסירה בטעות את כל ה-BGP Routes של Facebook מהאינטרנט.

תוך שניות, שרתי DNS של Facebook נעלמו מהאינטרנט. 3 מיליארד משתמשים לא יכלו להתחבר. הבעיה האמיתית: גם המהנדסים לא יכלו להיכנס לשרתים כדי לתקן — כי כלי הניהול הפנימיים גם הם רצו על אותה תשתית שנפלה. צוותים טסו פיזית למרכזי נתונים כדי לאפס ציוד ידנית. עלות: מאות מיליוני דולרים בהכנסות שאבדו ועוד 6 מיליארד דולר ירידה בשווי המניה ביום אחד.
💡DORA בפרקטיקה: חברות Elite מפרסות קוד לפרודקשן כמה פעמים ביום, Lead Time של שעות, ו-MTTR (Mean Time to Restore) של פחות משעה. חברות Low Performers מפרסות פעם בחודשיים ומשחזרים אחרי יותר משבוע. DevOps הוא לא כלי — הוא השפה שמייצרת את הפער הזה.

מעבדה 1 — Docker: בנה ופרוס אפליקציה ראשונה
Dockerfile · Multi-stage · Compose · Debug · Non-root
EASY▶
בנה Node.js app, ארוז אותה ב-Docker עם Multi-stage Build, הרץ עם Compose כולל DB, ולמד לדבג קונטיינרים.
01כתוב Dockerfile עם Multi-stage Build ל-Node.js app. שלב 1: Build. שלב 2: Runtime עם non-root user.
02הוסף docker-compose.yml עם app + PostgreSQL + Volume לפרסיסטנסי.
03הרץ docker compose up --build ובדוק שה-app מדבר עם ה-DB.
04כנס לקונטיינר: docker exec -it <id> sh — חקור File System, בדוק Processes.
05גרום לאפליקציה לקרוס — קרא docker logs --tail 50 <id> ו-docker inspect <id>.
$ docker build -t myapp:v1 . $ docker run -p 3000:3000 --name test myapp:v1 $ docker exec -it test sh $ docker stats # CPU + Memory בזמן אמת
✅ תשובות — Docker
01
Dockerfile Multi-stage:
שלב Builder: FROM node:20-alpine AS builder → npm ci → npm run build
שלב Runtime: FROM node:20-alpine → adduser app → COPY --from=builder /app/dist → USER app → CMD ["node","dist/index.js"]
תוצאה: Image של ~120MB במקום ~1GB. הNode_modules של Dev לא נכנסים לRuntime Image.
02
docker-compose.yml:
services:
app: { build: ., depends_on: [db], environment: [DB_HOST=db] }
db: { image: postgres:15, volumes: [pgdata:/var/lib/postgresql/data] }
volumes: { pgdata: {} }
depends_on מבטיח שDB עולה לפני App — אבל לא מבטיח שPostgres מוכן. הוסף healthcheck!
05
דיבוג קריסה: docker logs --tail 100 <id> — ראה Error. docker inspect <id> | jq '.[0].State' — ראה ExitCode ו-OOMKilled. ExitCode 137 = killed by OOM Killer (אזל זיכרון).
💡 Non-root חובה: Docker רץ כ-root בברירת מחדל. אם קוד זדוני מתבצע בקונטיינר, הוא יכול לצאת מה-Container בתנאים מסוימים. USER app לפני CMD = Privilege Escalation הרבה יותר קשה.

מעבדה 3 — GitHub Actions CI/CD Pipeline

EASY▶
בנה Pipeline מלא: בדיקות אוטומטיות → Build Docker Image → Deploy לStaging → Approval ידני → Deploy לProduction.
01צור .github/workflows/ci-cd.yml עם trigger על push ל-main ועל PRs.
02Job 1 — test: הרץ Unit Tests עם Coverage Report. Job נכשל אם Coverage < 80%.
03Job 2 — build: Build ו-Push Docker Image עם Tag לפי github.sha.
04Job 3 — deploy-staging: Deploy אוטומטי לStagting Environment.

✅ תשובות — GitHub Actions
02
Coverage Gate:
- run: npm test -- --coverage --coverageThreshold='{"global":{"lines":80}}'
כשהTest Runner יחזיר Exit Code 1 אם Coverage נמוכה מ-80% — Job נכשל אוטומטית ו-Pipeline עוצר.
03
Docker Build ועם SHA Tag:
- name: Build and push
  uses: docker/build-push-action@v5
  with:
    push: true
    tags: myorg/myapp:${{ github.sha }},myorg/myapp:latest
SHA Tag = Immutable. תמיד יודעים איזה Commit מריץ בProduction.
05
Environment + Approval: ב-GitHub Settings → Environments → production → הוסף Required Reviewers. כשJob מגיע לstep עם environment: production — Pipeline מחכה לאישור בממשק GitHub.
⚠️ Secrets בPipeline: לעולם אל תעשו echo $SECRET. השתמשו ב-echo "::add-mask::${{ secrets.MY_SECRET }}" בתחילת הWorkflow — GitHub ימסך אוטומטית את הערך בכל הLogs.

בנה Infrastructure מלא ב-AWS עם Terraform: VPC, Subnets, EC2, S3, ו-Remote State ב-S3 עם Lock ב-DynamoDB.
01צור main.tf עם AWS Provider. הגדר Remote State Backend עם S3 + DynamoDB.
02כתוב Module ל-VPC: Public ו-Private Subnets, Internet Gateway, Route Tables.
03הוסף EC2 Instance ב-Public Subnet עם Security Group שמאפשר רק פורט 443.
04terraform plan -out=tfplan — בדוק מה הולך להיווצר לפני Apply.
05הוסף lifecycle { prevent_destroy = true } ל-DB Resource. נסה terraform destroy — מה קורה?
✅ תשובות — Terraform
01
Remote State Backend:
terraform { backend "s3" { bucket = "my-tfstate" key = "prod/terraform.tfstate" region = "us-east-1" dynamodb_table = "tf-lock" } }
DynamoDB נועל את ה-State כשמישהו מריץ Apply — מונע שני אנשים מלשנות Infrastructure במקביל ולגרום Conflict.
04
terraform plan: + = Resource חדש שייווצר. - = Resource שיימחק. ~ = Resource שישתנה. ראה -/+? זה Resource שייהרס וייווצר מחדש — DB שכזה = Downtime.
05
prevent_destroy: terraform destroy יחזיר שגיאה: "Instance cannot be destroyed". זה לא נעילת Cloud — זה הגנה ב-Terraform בלבד. AWS עצמו לא יודע על זה. אפשר לעקוף בהסרת הלייף-ציקל ואז Destroy.
💡 Rule of thumb: terraform plan תמיד לפני apply. terraform plan -destroy לפני destroy. תמיד תסתכל על מה הRedLine שנמחק. "Looks good to me" לחצת Apply בלי לקרוא = כרטיס Lottery.

מעבדה 5 — Prometheus + Grafana Monitoring Stack
Scraping · PromQL · Dashboard · Alerts · SLO
MEDIUM▶
הקם Stack מלא של Monitoring: Prometheus סורק Metrics, Grafana מציג Dashboard, Alertmanager שולח התראות.
01הרץ עם Docker Compose: Prometheus + Grafana + Node Exporter + App עם /metrics endpoint.
02כתוב prometheus.yml עם scrape_config לApp ול-Node Exporter.
03ב-Grafana: חבר Prometheus כ-Data Source, בנה Dashboard עם Request Rate, Error Rate, Latency p99.
04כתוב Alert Rule ב-Prometheus: Alert אם Error Rate עולה על 1% למשך 2 דקות.
05">גרום לApp לשלוח שגיאות — ראה שהAlert מופעל ב-Alertmanager.
✅ תשובות — Prometheus + Grafana
02
prometheus.yml בסיסי:
scrape_configs:
- job_name: 'myapp'
  static_configs: [{targets: ['app:3000']}]
- job_name: 'node'
  static_configs: [{targets: ['node-exporter:9100']}]
כל 15 שניות (ברירת מחדל) Prometheus יסרוק /metrics מכל Target.
03
PromQL ל-Golden Signals:
Request Rate: rate(http_requests_total[5m])
Error Rate %: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) * 100
p99 Latency: histogram_quantile(0.99, rate(http_duration_seconds_bucket[5m]))
04
Alert Rule:
- alert: HighErrorRate
  expr: rate(http_errors_total[5m]) / rate(http_requests_total[5m]) > 0.01
  for: 2m
  labels: {severity: critical}
for: 2m = Alert מופעל רק אחרי 2 דקות רצופות — מונע Flapping על Spikes חולפים.
💡 The Four Golden Signals (Google SRE): Latency, Traffic, Errors, Saturation. אם אתם מנטרים רק 4 דברים — אלה הם. כל מערכת שניתן למדוד אותה לפי ה-4 האלה = מספיק בשביל SLO סביר.

מעבדה 6 — Blue/Green Deployment ב-Kubernetes
Zero Downtime · Traffic Switching · Rollback · Canary
MEDIUM▶
ממש Blue/Green Deployment שמאפשר Zero Downtime ו-Instant Rollback ב-Kubernetes.
01פרוס Blue Deployment עם version: v1 label. Service מצביע לBlue.
02פרוס Green Deployment עם version: v2 — בלי שינוי בService.
03בדוק שGreen בריא: kubectl rollout status deployment/myapp-green
04">עבור Traffic לGreen: שנה Service Selector מ-v1 ל-v2. בדוק שכל Traffic עובר.
05Rollback: שנה Selector חזרה ל-v1. לכמה זמן לוקח? (תשובה: שנייה)
✅ תשובות — Blue/Green
01
Blue Deployment:
metadata: { name: myapp-blue, labels: {version: v1} }
spec: { selector: { version: v1 }, replicas: 3 }
Service: selector: { app: myapp, version: v1 }
04
Traffic Switch — שנייה אחת:
kubectl patch service myapp -p '{"spec":{"selector":{"version":"v2"}}}'
K8s מעדכן iptables/eBPF rules. כל Connection חדש הולך לGreen. Connections קיימים נגמרים בBlue. Zero Dropped Requests.
05
Rollback: שנה חזרה ל-v1 — שנייה. השאר את Blue Deployment קיים לפחות שעה לפני שמוחקים אותו. Blue/Green vs Canary: Blue/Green = הכל או כלום. Canary = 5% Traffic לV2, גדל בהדרגה. Canary בטוח יותר, Blue/Green פשוט יותר.
💡 עלות Blue/Green: בזמן ה-Deployment יש פי 2 Pods — עלות ענן גדולה יותר. Canary Release פותר זאת: Pods ב-v2 מתוספים בהדרגה בזמן שv1 יורד. ArgoCD ו-Argo Rollouts מממשים Canary אוטומטית.

מעבדה 7 — Secrets Management עם HashiCorp Vault
Init · Unseal · KV Store · Dynamic Secrets · K8s Auth
HARD▶
הגדר Vault, שמור Secrets, ממש Dynamic DB Credentials שפגות אוטומטית, וחבר ל-Kubernetes.
01vault server -dev — Server פיתוח. לProduction: vault operator init + vault operator unseal × 3
02הפעל KV Secrets Engine: vault secrets enable -path=secret kv-v2
03שמור Secret: vault kv put secret/myapp db_password=s3cr3t
04הפעל Database Secrets Engine עם PostgreSQL. בקש Dynamic Credentials: vault read database/creds/my-role
05חבר ל-Kubernetes: הפעל Kubernetes Auth Method ב-Vault. הגדר Vault Agent Injector ב-K8s.
✅ תשובות — HashiCorp Vault
01
Production Init:
vault operator init מחזיר 5 Unseal Keys + Root Token. שמור בנפרד (שלושה אנשים שונים אוחזים 3 מ-5 Keys = Shamir Secret Sharing). אסור לאחסן Root Token — השתמש בו פעם אחת ל-Admin ואז בטל.
04
Dynamic Credentials: Vault יוצר User ב-PostgreSQL עם Lease Time (נניח שעה). אחרי שעה — Vault מוחק את ה-User אוטומטית. אם Credentials נגנבו, הם פגים. זה ההבדל מSecurity בין Static Secrets לDynamic Secrets.
05
K8s Auth + Vault Agent: Pod מקבל Annotation: vault.hashicorp.com/agent-inject-secret-config: "secret/myapp". Vault Agent (Sidecar) מתחבר ל-Vault ומכניס Secret לקובץ בVolume. App קורא מהקובץ. App לא יודע ולא מחזיק Vault Token.
⚠️ -dev Mode: Vault בDev Mode שומר הכל בזיכרון ומאתחל בכל Restart. מעולה לפיתוח, אסור בProduction. בProduction: Storage Backend (Raft/Consul), TLS, HA Mode עם 3 Nodes לפחות.
