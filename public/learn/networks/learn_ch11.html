איך הכל מתחבר
מה קורה כשגולשים ל-Facebook? מסע אחד מלא — דרך כל חמש השכבות, מ-DHCP ועד HTTP 200 OK.

כל שלב = פרוטוקול אחד. יחד = גלישה לאינטרנט.
1
DHCP — Layer 3/4
קבלת כתובת IP
המחשב מתחבר לרשת — הוא יודע רק את כתובת ה-MAC שלו. הוא שולח DHCP Discover ב-Broadcast (src=0.0.0.0, dst=255.255.255.255). שרת DHCP עונה עם Offer → Request → ACK. המחשב מקבל:
כתובת IP שלנו:192.168.0.5
Subnet Mask:255.255.0.0
Default Gateway:192.168.0.1
DNS Server:2.2.2.2
2
ARP — Layer 2
גילוי MAC של הנתב
DNS Server (2.2.2.2) לא ב-Subnet שלנו (192.168.x.x) → חייבים לעבור דרך Default Gateway. אבל Ethernet frame דורש MAC יעד. שולחים ARP Request ב-Broadcast: "מי יש את MAC של 192.168.0.1?"
ARP Request:dst=ff:ff:ff:ff:ff:ff (Broadcast)
שואלים:"מי 192.168.0.1?"
ARP Reply:נתב: "אני! MAC=aa:bb:cc:dd:ee:ff"
Switch:למד MAC שלנו → Port X בזמן DHCP Discover
3
DNS — Layer 5 (over UDP)
מציאת IP של Facebook
שולחים שאילתת DNS מעל UDP לכתובת 2.2.2.2. כל נתב בדרך מוריד TTL ב-1 ומעדכן Checksum. כתובות MAC משתנות בכל Hop — כתובות IP נשארות קבועות!
שאילתא:A record: www.facebook.com?
Transaction ID:1337
תשובה:www.facebook.com → 3.3.3.3
פורט src/dst:random / 53
4
TCP — Layer 4
Three-Way Handshake עם Facebook
עכשיו יש לנו IP של Facebook (3.3.3.3). לפני HTTP — חייבים להרים קישור TCP. הלקוח מגריל ISN=333, שרת Facebook מגריל ISN=500.
פורט dst (Facebook):80 (HTTP)
פורט src (מחשב):60124 (random, 49152-65535)
SYN:Seq=333, ACK=0
SYN+ACK:Seq=500, ACK=334
ACK:Seq=334, ACK=501
5
HTTP — Layer 5
שליחת בקשת GET
הקישור קם. עכשיו שולחים בקשת HTTP GET לעמוד הראשי של Facebook.
בקשה:GET / HTTP/1.0
Host:www.facebook.com
תשובה:HTTP/1.0 200 OK
Content:HTML של דף הבית 🎉
6
NAT — ברשתות פרטיות
המחשב מאחורי NAT
ברוב הבתים: IP של המחשב = פרטי (192.168.x.x). Facebook לא יכול לשלוח ישירות לכתובת פרטית. הנתב מחליף כתובות בדרך.
יציאה:נתב מחליף src: 192.168.0.5 → 1.1.1.1
Facebook רואה:בקשה מ-1.1.1.1 בלבד
כניסה:נתב מחליף dst: 1.1.1.1 → 192.168.0.5
מחשב רואה:תשובה כאילו ישירות!
✓
🎉 סיום
הדפדפן מציג את Facebook!
הדפדפן קיבל HTTP 200 OK עם HTML. המסע הסתיים. כל זה קרה בפחות משנייה. DHCP → ARP → DNS → TCP Handshake → HTTP GET → 200 OK.
ניתוח Hops
כתובות MAC ו-IP בכל Hop

המחשב (192.168.0.5) שולח DNS Query ל-2.2.2.2 דרך נתבים:

HopMAC srcMAC dstIP srcIP dstTTL
מחשב → נתב MAC_מחשב MAC_נתב_LAN 192.168.0.5 2.2.2.2 128
נתב → R1 MAC_נתב_WAN MAC_R1 192.168.0.5 2.2.2.2 127
R1 → R2 MAC_R1_out MAC_R2 192.168.0.5 2.2.2.2 126
R2 → DNS MAC_R2_out MAC_DNS 192.168.0.5 2.2.2.2 125
⚠️ MAC משתנה בכל Hop! IP נשאר קבוע לכל אורך הדרך. Switch לא משנה TTL. רק נתבים משנים TTL.

סיכום מלא
כל פרוטוקול במסע
DHCP
Layer 3/4 — UDP
מקצה IP, Mask, Gateway, DNS. 4 הודעות: Discover→Offer→Request→ACK. Broadcast!
ARP
Layer 2 — Ethernet
ממפה IP → MAC. Request=Broadcast, Reply=Unicast. לא עובר נתבים. Cache מונע שאילתות חוזרות.
DNS
Layer 5 — UDP port 53
ממיר שם דומיין ל-IP. Transaction ID מזהה שאלה-תשובה. שאילתא מסוג A.
TCP
Layer 4 — מבוסס קישור
Three-Way Handshake. Seq/ACK מספרים. ISN רנדומלי. פורט dst=80, src=49152-65535.
HTTP
Layer 5 — over TCP
GET / HTTP/1.0 לבקשה. 200 OK לתשובה. על גבי קישור TCP פתוח.
NAT
ברשתות פרטיות
נתב מחליף src IP ביציאה, dst IP בכניסה. שקוף למחשב ולשרת. מאפשר IP פרטי.
Switch
Layer 2 — MAC Table
לומד MAC→Port. שולח רק ליעד. לא משנה TTL. Broadcast=Flood תמיד.
Router
Layer 3 — Routing Table
מוריד TTL ב-1. מעדכן Checksum. מחליף MAC לפי hop. IP נשאר קבוע.
