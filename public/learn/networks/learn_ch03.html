חלק א' — מדוע מודל שכבות?

🏗️ הבעיה שהמודל פותר

בעבר, כל יצרן חומרה פיתח פרוטוקול תקשורת קנייני משלו. כתוצאה מכך, מכשירים של יצרנים שונים לא יכלו לתקשר — לקוחות היו מחויבים לאותה יצרנית לכל הציוד שלהם.

ארגון ISO יצר את מודל OSI (7 שכבות) ובהמשך נוצר מודל TCP/IP המעשי שבו אנו משתמשים — מודל חמש השכבות.

עיקרון המפתח

כל שכבה מספקת שירות לשכבה שמעליה מבלי לחשוף כיצד השירות ממומש. שינוי מימוש שכבה אחת לא שובר את שאר המערכת — "קופסה שחורה".

אנלוגיה מהפרק — שדה תעופה

שכבת המטוס לא יודעת ולא מתעניינת אם הנוסע הגיע לשדה התעופה ברכב, אוטובוס או רגלית. היא אחראית על הטיסה בלבד. כך כל שכבה ברשת.

חלק ב' — חמש השכבות

📊 טבלת חמש השכבות

# שם מטרה פרוטוקול לדוגמה שם יחידת מידע
5 Application שימושי אפליקציה HTTP, SMTP, FTP Message

חשוב לזכור

Sockets שלמדנו בפרק 2 הם API של מערכת ההפעלה שפועל בשכבת האפליקציה. הפרמטרים שנותנים (IP + Port) מתייחסים לשכבות 3 ו-4 שמתחת.

📦 Encapsulation — כימוס

תהליך השליחה (Encapsulation)

כל שכבה מוסיפה Header בראש הנתונים שמגיעים מהשכבה שמעליה:

[App Data]
[TCP Header] + [App Data]
[IP Header] + [TCP Header] + [App Data]
[Ethernet Header] + [IP Header] + [TCP Header] + [App Data] + [Ethernet Trailer]
→ bits על הכבל

תהליך הקבלה (Decapsulation)

ביעד קולפים שכבה אחר שכבה מלמטה למעלה — כמו חבילה עוברת שמורידים ממנה עטיפות.

עיקרון חשוב

ה-Data של שכבה N = הפקטה השלמה של שכבה N+1. כשאומרים "Frame" — הכוונה לשכבה 2 + כל השכבות שמעליה בפנים.

🗣️ פרוטוקול

פרוטוקול = הסכם שמחייב שני צדדים לסט כללים קבוע על פורמט התקשורת.

דוגמאות מהפרקים
  • הפרוטוקול שכתבתם בתרגיל 2.6 — כל פקודה בדיוק 4 בתים (TIME, NAME, RAND, EXIT)
  • HTTP — פרוטוקול גלישה, שולט כיצד דפדפן מדבר עם שרת
  • SMTP — פרוטוקול מייל

למה חולקים למידע לפקטות?

  • בקרת שגיאות: שגיאה? שולחים רק את הפקטה הפגומה, לא הכל
  • שיתוף קו: כמה תוכניות/מחשבים שולחים לסירוגין — כל אחד שולח פקטה ומפנה
  • סנכרון חומרה: קל יותר לסנכרן בין ישויות ברשת
חלק ג' — Wireshark בפועל

🦈 מבנה חלון Wireshark

4 חלקי המסך הראשי
  • אדום — Display Filter: סינון תצוגה
  • כחול — רשימת פקטות: No / Time / Source / Destination / Protocol / Length / Info
  • ירוק — ניתוח פקטה לשכבות
  • כתום — תוכן גולמי: Hex + ASCII

Promiscuous Mode (מצב פרוץ)

כרטיס רשת בדרך כלל זורק מסגרות לא מיועדות אליו. Promiscuous Mode מורה לו לקלוט כל מסגרת שרואה — גם תעבורה של מחשבים אחרים ברשת.

🔍 Display Filters — דוגמאות מהפרק

פילטר מה הוא מציג
icmp פקטות ICMP בלבד (ping)
tcp כל פקטות TCP
ip.src == 192.168.1.1 פקטות שיצאו מכתובת זו בלבד
ip.dst == 192.168.1.1 פקטות שהולכות לכתובת זו בלבד
ip.addr == 192.168.1.1 פקטות שבהן כתובת זו היא מקור OR יעד
tcp.port == 80 פקטות TCP עם פורט 80 (מקור OR יעד)
ip.addr == X and tcp.port == 443 שילוב עם and / or
frame contains "txtPa" מסגרות שמכילות טקסט מסוים (תרגיל 3.2)
icmp.resptime > 100 שדה מחושב ע"י Wireshark, לא בפקטה עצמה
Display Filter לעומת Capture Filter

Display Filter = תצוגה בלבד, פקטות שסוננו עדיין קיימות, ניתן לשנות תוך כדי. Capture Filter = פועל ב-Driver, פקטות שסוננו לא נקלטות בכלל, מהיר יותר.

🔓 תרגיל 3.2 — הסנפת סיסמה

נכנסים לאתר HTTP רגיל (לא HTTPS), ממלאים פרטים ומסניפים. עם הפילטר frame contains "txtPa" מוצאים פקטה שמכילה את הסיסמה בטקסט גלוי!

המסקנה הביטחונית

כל מה שנשלח ב-HTTP עובר בגלוי ברשת. כיום רוב האתרים משתמשים ב-HTTPS = HTTP + TLS שמצפין את כל ה-Stream כולל URL, headers וגוף. לעולם לא להזין סיסמה אמיתית באתרי HTTP.
