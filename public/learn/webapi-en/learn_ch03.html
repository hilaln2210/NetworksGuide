REST API — Design Done Right
Resources, Naming, Versioning, Pagination, Idempotency, RFC 7807
REST⏱ 60m▶
HTTP is a protocol — a language. REST is an architectural style — rules for designing your API on top of HTTP. Most APIs you'll encounter (GitHub, Stripe, Twitter) are REST APIs. But a poorly designed REST API is one of the most frustrating things.
🗂️ REST Like a Well-Organized Library:
A library has Resources (books, authors, categories) — each with a clear address. You don't say "give me John's science books" — you go to /authors/john/books?genre=science. The URL describes what you're looking for, the Method describes what you're doing.
Rule 1 — Resource Naming
❌
Wrong naming (RPC-style):
/getUser?id=42
/createNewOrder
/deleteUserAccount/5
The URL describes an action — this is not REST.
✅
Correct naming (Resource-style):
GET /users/42 — read user
POST /orders — create order
DELETE /users/5 — delete user
The URL describes an entity, the Method describes an action.
Rule 2 — Idempotency
🔁 Idempotency = "safe to send twice without side effects":
Imagine you clicked "Pay" in a food app and the network dropped — unknown if the order went through. Is it safe to click again?

GET, PUT, DELETE — yes. Send as many times as you want — same result.
POST — no! Send twice = two orders. That's why Stripe uses Idempotency Keys.
Rule 3 — Versioning
Why?
An API that people use can't "just change" — 10,000 apps would break. You need versions.
How?
URL (most common): /api/v1/users → /api/v2/users
Header: Accept: application/vnd.api+json;version=2
Query Param: /api/users?version=2
Rule 4 — Pagination
Problem
What if there are a million users? Don't return everything — return pages.
OFFSET
GET /users?limit=20&offset=40 — "give me 20 starting at row 40"
Problem: if a user is added during scrolling — you'll miss a row. Heavy on the DB for large tables.
Cursor
GET /users?limit=20&after=user_abc123 — "give me 20 after this user"
Better: faster, no missed data. This is what Twitter/Instagram use.
💡 RFC 7807 — Standard Error Response: Instead of returning {"error": "bad"} — return: {"type": "...", "title": "Validation Failed", "status": 400, "detail": "email is required"}. Every Client can handle errors in a consistent way.
Real Incident
🔴 BREAK — 2018 / Twitter
When Twitter Killed Millions of Apps in One Day
In 2013, Twitter announced the deprecation of API v1 for 2018. 5 years seemed like a lot — but 2018 arrived and many apps crashed.

The problem: no Sunset Header telling every Client "I'm closing on date X." Developers who didn't follow the blog learned the news when their service went down.

The lesson that became a standard: Sunset: Sat, 01 Jan 2018 00:00:00 GMT in every Response + 6 months of parallel support.
📖 Glossary — This Chapter ▶
RESTRepresentational State Transfer — an architectural style for APIs. Not a language, not a protocol — a set of rules.
ResourceAn entity in the API — user, order, product. Represented by a unique URL.
EndpointA specific address in the API. /users is one Endpoint, /users/42 is another.
IdempotencyAn operation you can repeat and get the same result. GET/PUT/DELETE are Idempotent, POST is not.
PaginationReturning data in pages instead of all at once. OFFSET-based or Cursor-based.
VersioningAPI version management — v1, v2. Allows changing the API without breaking existing Clients.
RFC 7807Standard for API error message format. Defines fields: type, title, status, detail.
Sunset HeaderHTTP Header announcing when an API will close: Sunset: Sat, 01 Jan 2025 00:00:00 GMT
