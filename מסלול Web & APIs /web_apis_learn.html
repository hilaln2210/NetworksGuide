מה זה API בכלל? — מבוא מאפס
לגמרי מהתחלה · ללא ידע מוקדם · אנלוגיות מהחיים
התחלה ⏱ 25m ▶
לפני שנדבר על HTTP, REST, JSON ו-Endpoints — צריך לענות על שאלה אחת: מה בעצם קורה כשאפליקציה "מדברת" עם שרת?

אם אתה פותח Waze, מזמין ב-Wolt, או מתחבר עם Google לאפליקציה אחרת — כל אחד מהרגעים האלה מערב API. הפרק הזה מסביר מה זה מהתחלה, ללא ידע מוקדם.
מה זה API?
🍽️ האנלוגיה של המסעדה:
דמיין מסעדה. אתה (הלקוח) יושב בחדר האוכל. המטבח הוא השרת. אתה לא נכנס למטבח — אתה קורא למלצר.

ה-API הוא המלצר.
אתה מזמין (Request), המלצר הולך למטבח, ומביא לך את המנה (Response). אתה לא יודע ולא צריך לדעת איך הבישול עובד.
01
API = Application Programming Interface = ממשק שמאפשר לשתי תוכנות לדבר אחת עם השנייה. זה לא קוד מיוחד — זו הסכמה: "שלח לי בקשה בפורמט הזה, ואני אחזיר לך תשובה בפורמט הזה."
02
דוגמה מהחיים: כשאתה לוחץ "Login with Google" באפליקציה — האפליקציה שולחת בקשה ל-API של Google, Google בודקת מי אתה, ומחזירה: "זה יוסי, המייל שלו הוא [email protected]". האפליקציה קיבלה מידע בלי לדעת את הסיסמה שלך.
03
למה זה חשוב? כמעט כל אפליקציה מודרנית בנויה מ-APIs — Waze קונה API של מפות מגוגל, Wolt מחבר בין API של מסעדות, API של תשלומים, API של מיקום. ה-APIs הם "הצינורות" שמחברים הכל.
Client ו-Server — מי מדבר עם מי?
📞 אנלוגיית הטלפון:
Client = מי שמתקשר (שואל שאלות, שולח בקשות)
Server = מי שעונה (מחזיק מידע, מעבד בקשות)

הדפדפן שלך הוא Client. כשאתה פותח youtube.com — הדפדפן מתקשר לשרתי YouTube ומבקש: "תן לי את הדף הראשי". השרת עונה עם HTML, תמונות, ווידאו.
💡 נקודה חשובה: Client ו-Server יכולים להיות כל דבר. אפליקציית מובייל היא Client. דפדפן הוא Client. אפילו שרת אחד יכול להיות Client של שרת אחר — כשהם מדברים ביניהם.
HTTP — שפת התקשורת של האינטרנט
אם API הוא המלצר, אז HTTP הוא השפה שבה הם מדברים. כשהדפדפן שלך מבקש דף — הוא שולח הודעה בפורמט HTTP. השרת עונה ב-HTTP.
שאל
Request (בקשה) — Client שולח לשרת:
GET /api/users/42 — "תן לי את פרטי המשתמש מספר 42"
POST /api/orders — "צור הזמנה חדשה עם הנתונים הבאים"
DELETE /api/posts/7 — "מחק את הפוסט מספר 7"
קבל
Response (תשובה) — Server מחזיר:
200 OK — הצליח, הנה הנתונים
404 Not Found — לא מצאתי את מה שביקשת
500 Server Error — משהו השתבש אצלי
🔢 Status Codes — ספרות שמספרות את הסיפור:
200-299 = ✅ הצליח
300-399 = ↗️ הועבר לכתובת אחרת
400-499 = ❌ טעית בבקשה (הלקוח אשם)
500-599 = 💥 שגיאת שרת (השרת אשם)

בדיוק כמו ציונים בבית ספר — המספר מספר לך בלי לקרוא הרבה אם זה עבר או נכשל.
JSON — פורמט הנתונים
שלחנו בקשה, קיבלנו תשובה — אבל באיזה פורמט מגיעים הנתונים? כמעט תמיד ב-JSONJSONJavaScript Object Notation — פורמט טקסטואלי פשוט לייצוג נתונים. קריא לבני אדם ולמחשבים..
מה
JSON נראה כך:
{ "name": "Dana", "age": 28, "city": "Tel Aviv" }

זה פשוט טקסט מאורגן. מפתח (name) ואחריו ערך ("Dana"). כל API שאתה מדבר איתו יחזיר לך JSON כזה עם הנתונים שביקשת.
למה
למה JSON ולא משהו אחר?
כי כל שפת תכנות יכולה לקרוא אותו. Python, JavaScript, Java — כולם "מדברים" JSON. זה הפך לשפת האם של ה-Web.
הכל יחד — מה קורה כשאתה פותח Wolt?
1
האפליקציה (Client) שולחת בקשת HTTP לשרת של Wolt: GET /api/restaurants?city=tlv
2
שרת Wolt (Server) מקבל את הבקשה, שולף מסד הנתונים את המסעדות בתל אביב
3
השרת מחזיר JSON: [{"name": "בורגר סאלים", "rating": 4.8}, ...]
4
האפליקציה קוראת את ה-JSON ומציגה את המסעדות על המסך
🎯 הרגע שבו הכל מתחבר: כל הפרקים הבאים מרחיבים על שלב אחד מהתהליך הזה. HTTP → REST → Auth → WebSockets. אבל הבסיס הוא תמיד: Client שואל, Server עונה, JSON מחזיר נתונים.

HTTP — הפרוטוקול שמריץ את האינטרנט
Request, Response, Headers, Methods, Status Codes, HTTP/2, HTTP/3
CORE⏱ 50m▶
כל פעם שאתה פותח אתר, שולח הודעה, או מחפש ב-Google — הדפדפן שלך שולח בקשת HTTP לשרת מרוחק. HTTP הוא השפה הבסיסית שבה כל הרשת מדברת. לפני שנבין API — חייבים להבין את השפה עצמה.
📬 HTTP כמו מערכת דואר מאוד מסודרת:
כל בקשה היא מעטפה עם כתובת מדויקת, סוג מסר (מכתב? חבילה? טופס?), ותוכן. כל תשובה מגיעה עם קוד שאומר: "נמסר ✅", "לא נמצא ❌", "תקלה בדואר 💥". אתה לא יוצר קשר ישיר עם הנמען — הכל עובר דרך המערכת.
Request — מה שולחים
Method
מה אתה רוצה לעשות?
GET — תן לי מידע (קריאה בלבד)
POST — צור משהו חדש
PUT — החלף משהו קיים במלואו
PATCH — עדכן חלק ממשהו קיים
DELETE — מחק משהו
URL
לאן? הכתובת המדויקת: https://api.example.com/users/42
https:// = פרוטוקול מאובטח
api.example.com = הדומיין / השרת
/users/42 = ה-Endpoint הספציפי
Headers
מטא-דאטה על הבקשה:
Content-Type: application/json — "שולח לך JSON"
Authorization: Bearer eyJ... — "הנה הטוקן שלי"
Accept: application/json — "אני מצפה לקבל JSON"
Body
התוכן עצמו (רק ב-POST/PUT/PATCH):
{"name": "Dana", "email": "dana@example.com"}
ב-GET אין Body — כל הנתונים ב-URL עצמו.
Status Codes — מה הייתה התשובה
🚦 5 קטגוריות, מספר אחד מספר את הכל:
1xx — מידע בלבד (נדיר)
2xx — ✅ הצליח (200 OK, 201 Created, 204 No Content)
3xx — ↗️ Redirect (301 Moved, 304 Not Modified)
4xx — ❌ הלקוח טעה (400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests)
5xx — 💥 השרת שגה (500 Internal Error, 503 Service Unavailable)
HTTP/1.1 → HTTP/2 → HTTP/3
1.1
HTTP/1.1 (1997) — בקשה אחת בכל פעם על כל חיבור. כמו תור לקופה — כולם מחכים לאחד.
2
HTTP/2 (2015) — כמה בקשות במקביל על אותו חיבור (Multiplexing). Header Compression. פי 2-3 מהיר.
3
HTTP/3 (2022) — רץ על UDP במקום TCP. אם Packet אבד — רק ה-Stream שלו מחכה, לא כולם. מצוין ב-Mobile.
💡 Statelessness — עיקרון מרכזי: כל בקשת HTTP עצמאית לחלוטין. השרת לא זוכר שביקרת לפני. לכן צריך לשלוח Token בכל בקשה — השרת לא "זוכר" שהתחברת.

REST API — תכנון שנעשה נכון
Resources, Naming, Versioning, Pagination, Idempotency, RFC 7807
REST⏱ 60m▶
HTTP הוא פרוטוקול — שפה. REST הוא סגנון ארכיטקטורה — כללים איך לתכנן את ה-API שלך על גבי HTTP. רוב ה-APIs שתפגוש (GitHub, Stripe, Twitter) הם REST APIs. אבל REST API גרוע הוא אחד הדברים הכי מתסכלים שיש.
🗂️ REST כמו ספרייה מסודרת:
בספרייה יש Resources (ספרים, מחברים, קטגוריות) — לכל אחד כתובת ברורה. אתה לא אומר "תן לי את ספרי המדע של ג'ון" — אתה הולך ל /authors/john/books?genre=science. ה-URL מתאר מה אתה מחפש, ה-Method מתאר מה אתה עושה.
כלל 1 — Resource Naming
❌
שמות שגויים (RPC-style):
/getUser?id=42
/createNewOrder
/deleteUserAccount/5
ה-URL מתאר פעולה — זה לא REST.
✅
שמות נכונים (Resource-style):
GET /users/42 — קרא משתמש
POST /orders — צור הזמנה
DELETE /users/5 — מחק משתמש
ה-URL מתאר ישות, ה-Method מתאר פעולה.
כלל 2 — Idempotency
🔁 Idempotency = "אפשר לשלוח פעמיים בלי נזק":
דמיין שלחצת "שלם" ב-Wolt והרשת נפלה — לא ידוע אם ההזמנה עברה. האם בטוח ללחוץ שוב?

GET, PUT, DELETE — כן. שלח כמה פעמים שרוצה — אותה תוצאה.
POST — לא! שלח פעמיים = שתי הזמנות. לכן Stripe משתמש ב-Idempotency Keys.
כלל 3 — Versioning
למה?
API שמשתמשים בו לא יכול "פשוט להשתנות" — 10,000 אפליקציות יישברו. צריך גרסאות.
איך?
URL (הנפוץ ביותר): /api/v1/users → /api/v2/users
Header: Accept: application/vnd.api+json;version=2
Query Param: /api/users?version=2
כלל 4 — Pagination
בעיה
מה קורה אם יש מיליון משתמשים? לא מחזירים הכל — מחזירים בדפים.
OFFSET
GET /users?limit=20&offset=40 — "תן לי 20 מהשורה ה-40"
בעיה: אם נוסף user בזמן הגלילה — תפסיד שורה. כבד ב-DB על טבלאות גדולות.
Cursor
GET /users?limit=20&after=user_abc123 — "תן לי 20 אחרי המשתמש הזה"
עדיף: מהיר יותר, לא מפספס נתונים. זה מה-Twitter/Instagram משתמשים בו.
💡 RFC 7807 — Error Response סטנדרטי: במקום להחזיר {"error": "bad"} — החזר: {"type": "...", "title": "Validation Failed", "status": 400, "detail": "email is required"}. כל Client יכול לטפל בשגיאות בצורה אחידה.

📖 מילון מושגים — פרק זה ▶
RESTRepresentational State Transfer — סגנון ארכיטקטורה ל-APIs. לא שפה, לא פרוטוקול — אוסף כללים.
Resourceישות ב-API — user, order, product. מיוצג על ידי URL ייחודי.
Endpointכתובת ספציפית ב-API. /users הוא Endpoint, /users/42 הוא Endpoint אחר.

Paginationחזרת נתונים בדפים במקום הכל בבת אחת. OFFSET-based או Cursor-based.
Versioningניהול גרסאות API — v1, v2. מאפשר לשנות API בלי לשבור Clients קיימים.

GQL
GraphQL — בקש רק מה שאתה צריך
Schema, Queries, Mutations, Subscriptions, N+1, DataLoader
GRAPHQL⏱ 70m▶
REST עם כל כלליו הוא מצוין — אבל יש בעיה: אתה מקבל מה שהשרת מחליט לתת לך. נניח שאתה צריך רק שם ותמונה של User — REST מחזיר 20 שדות. GraphQL פותר זאת: הClient מחליט בדיוק מה לקבל.
🍕 GraphQL כמו פיצה Build-Your-Own:
ב-REST: "הגבינה עם הפטריות" — תקבל את מה שהמסעדה מחליטה שנמצא בפיצה הזו.
ב-GraphQL: "אני רוצה בצק דק, עגבנייה, גבינה, ורק זיתים" — אתה מזמין בדיוק מה שצריך.

Facebook המציאה אותו ב-2012 כי ה-Mobile App שלהם היה איטי — מורידים יותר מדי נתונים.
Over-fetching ו-Under-fetching
❌
Over-fetching ב-REST: צריך רק name ו-avatar של User — מקבל 15 שדות כולל address, preferences, metadata...
❌
Under-fetching ב-REST: רוצה User + Orders שלו + פרטי Product בכל Order — צריך 3 בקשות נפרדות.
✅
GraphQL: בקשה אחת, מקבל בדיוק מה שביקשת:
query {"{"} user(id:"42") {"{"} name avatar orders {"{"} total {"}"} {"}"} {"}"}
N+1 Problem — הבעיה הכי נפוצה ב-GraphQL
⚠️ מה זה N+1 ולמה זה מסוכן:
בקשת 10 Users עם ה-Orders שלהם = 1 שאילתת DB לUsers + 10 שאילתות לOrders = 11 שאילתות.
בקשת 100 Users = 101 שאילתות. 1000 = 1001.

הפתרון: DataLoader — מרכז את כל הבקשות ל-Orders וב-Batch אחד: 1+1=2 שאילתות בלבד.
מתי GraphQL ומתי REST?
REST
עדיף כשמה: Public API פשוט, Caching חשוב, הצוות מכיר REST, CRUD בסיסי
GraphQL
עדיף כשמה: כמה Clients עם צרכים שונים (Web/Mobile), נתונים מורכבים ומקוננים, הצוות מנוסה
אירוע אמיתי
🔴 OVERLOAD — 2019 / Shopify
שאילתת GraphQL אחת שהפילה את בסיס הנתונים
ללא הגבלת עומק שאילתות, מפתח כתב: user → friends → friends → orders → items → reviews (עומק 7).

100 בקשות כאלה = מיליוני שאילתות DB = קריסה.

הפתרון של Shopify: Depth Limit=5, Complexity Score לכל שאילתה, Rate Limiting לפי עלות השאילתה.

Over-fetchingקבלת יותר נתונים ממה שצריך. בזבוז Bandwidth ומהיר פחות.
Under-fetchingצריך כמה בקשות כדי לאסוף את כל הנתונים הדרושים.
Schemaהגדרת כל הסוגים, שדות, ושאילתות אפשריות ב-GraphQL API.
Queryבקשת קריאה ב-GraphQL — כמו GET ב-REST.
Mutationבקשת כתיבה ב-GraphQL — כמו POST/PUT/DELETE ב-REST.
N+1 Problemבעיית ביצועים: שאילתת N רשומות = N+1 שאילתות DB. פותר עם DataLoader (Batching).
DataLoaderספרייה שמרכזת שאילתות לBatch אחד. מ-N+1 ל-1+1.

AUTH
Authentication & Authorization
JWT, OAuth 2.0, OpenID Connect, PKCE, API Keys, CSRF
SECURITY⏱ 80m▶
עד כאן למדנו לשלוח בקשות ולקבל נתונים — אבל מי מורשה לשאול? Authentication ו-Authorization הם שני מנגנונים שנשמעים דומים אבל שונים לחלוטין. בלעדיהם, כל API הוא פתוח לכולם.
🏢 כניסה לבניין משרדים:
Authentication = מי אתה? (תעודת זהות, טביעת אצבע, סיסמה)
Authorization = מה מותר לך? (רק לקומה 3, לא לחדר שרתים)

אפשר להיות מאומת (Authentication ✅) ועדיין חסום (Authorization ❌).
JWT — הToken הנפוץ ביותר
מה?
JWT = JSON Web Token — טוקן מקודד עם 3 חלקים מופרדים בנקודה:
eyJhbGci.eyJzdWIiOiI0Mn0.SflKxwRJSMeK
מבנה
Header — אלגוריתם ההצפנה (HS256/RS256)
Payload — גלוי לכולם! מכיל user_id, roles, תפוגה
Signature — חתימה שמאמתת שה-Token לא שונה
⚠️
Payload גלוי לכולם — Base64 בלבד, לא הצפנה. אל תשים סיסמאות ב-Payload!
OAuth 2.0 — Login with Google/GitHub
🔑 OAuth 2.0 כמו מפתח וולט:
במקום לתת למישהו את מפתח הבית — נותן לו מפתח וולט מוגבל שפותח רק את הדלת הקדמית לשעה.

כשאתה לוחץ "Login with Google": האפליקציה מקבלת Token מוגבל שמאפשר לה לקרוא את ה-Email שלך — לא את הסיסמה שלך ב-Google.
1
User לוחץ "Login with Google" → אפליקציה מפנה ל-Google
2
User מאשר ל-Google לשתף את הEmail עם האפליקציה
3
Google מחזיר Authorization Code לאפליקציה
4
אפליקציה מחליפה Code ב-Access Token → משתמשת אותו לקרוא Email

Authenticationאימות זהות — מי אתה? סיסמה, ביומטריה, Token.
Authorizationהרשאה — מה מותר לך? Roles, Permissions, Scopes.

Scopeהגדרת מה ה-Token מאפשר: read:email, write:posts וכד'.

WS
WebSockets & Real-Time
WebSocket, SSE, Long Polling, WebRTC, Socket.io, Scaling
REAL-TIME⏱ 60m▶
HTTP הוא Request-Response: אתה שואל, השרת עונה, החיבור נסגר. אבל מה אם השרת צריך לדחוף לך עדכון — הודעה חדשה בWhatsApp, מניה שמשתנה, מיקום של Waze? לא יכול לחכות שתשאל. כאן נכנסים WebSockets ו-SSE.
📞 HTTP vs WebSocket:
HTTP = SMS. שולח הודעה, מקבל תשובה, שיחה נגמרת. לכל שאלה — SMS חדש.
WebSocket = שיחת טלפון. חיבור פתוח, שני הצדדים מדברים ברציפות, עד שמישהו מנתק.

WebSocket מתחיל כ-HTTP, ואז "מתרומם" לחיבור דו-כיווני מתמשך.
WebSocket Handshake — איך זה עובד
1
Client שולח בקשת HTTP רגילה עם Header מיוחד: Upgrade: websocket
2
Server מאשר: מחזיר 101 Switching Protocols
3
החיבור TCP נשאר פתוח. עכשיו שני הצדדים שולחים Messages מתי שרוצים — ללא בקשה חדשה.
SSE — כשצריך רק חד-כיוון
📻 SSE כמו רדיו:
Server שולח, Client מקבל — חד-כיווני. Client לא יכול לשלוח חזרה על אותו Channel.
מתי SSE עדיף על WebSocket? Notifications, Live Feed, Dashboard Updates — כשרק צריך לקבל עדכונים, לא לשלוח.
יתרון: פשוט יותר, Auto-Reconnect מובנה, עובד על HTTP/2.
אירוע אמיתי
🔴 OUTAGE — 2020 / Discord
הקורונה שבר את Discord תוך שבוע
COVID הביא גידול מ-2M ל-5M Concurrent WebSocket Connections תוך ימים. גידול שתוכנן לשנים קרה בשבוע.

WebSocket שונה מ-HTTP: כל משתמש מחזיק חיבור פתוח ומתמשך. 5M חיבורים פתוחים = עומס אחר לגמרי.

הפתרון: Redis Pub/Sub — כל שרת WebSocket מאזין לאירועים דרך Redis, כך מסר מ-Server A מגיע ל-User שמחובר ל-Server B.
📖 מילון מושגים — פרק זה ▶
WebSocketפרוטוקול חיבור דו-כיווני מתמשך. מתחיל כ-HTTP ומתרומם ל-Full-Duplex.
SSEServer-Sent Events — דחיפה חד-כיוונית מServer ל-Client. פשוט, Auto-Reconnect.
Long PollingClient שולח בקשה, Server "תולה" אותה עד שיש חדשות. פחות יעיל מ-WebSocket.
Handshakeתהליך פתיחת חיבור WebSocket — HTTP Upgrade Request + 101 Response.
Full-Duplexשני הצדדים שולחים ומקבלים בו-זמנית. שיחת טלפון היא Full-Duplex, SMS אינו.
Redis Pub/Subמנגנון publish-subscribe ב-Redis. מאפשר ל-N שרתים לשתף אירועים Real-Time.
Sticky SessionsLoad Balancer שמוודא שאותו User תמיד מגיע לאותו Server — חשוב ל-WebSocket.
⚡
Performance & Caching
CDN, Cache-Control, ETags, Rate Limiting, Compression, Connection Pool
PERF⏱ 70m▶
הAPI שלך עובד — אבל כמה מהר? ו-כמה עולה? Caching הוא הדרך לענות על שאלות שכבר נשאלו בלי לעבוד מחדש. Cache-Control הוא ה-Header שקובע מה, כמה, ואיך.
🗂️ Cache כמו פתק הליד:
במקום לשאול כל פעם "מה מספר הטלפון של עמיר?" — כתבת אותו על פתק. בפעם הבאה — קרא מהפתק.
אבל אם עמיר שינה מספר ואתה לא יודע — הפתק מוטעה. לכן צריך להגדיר: "הפתק הזה תקף X דקות".
Cache-Control — Header שמגדיר הכל
max-age
Cache-Control: max-age=300 — "שמור 5 דקות, אל תשאל שוב"
המשתמש לא מחכה, Cache עונה מיידית. אחרי 5 דקות — בקשה חדשה.
no-cache
Cache-Control: no-cache — "שמור, אבל בדוק תמיד אם עדכני"
לא אומר "אל תשמור" — אומר "אמת לפני שימוש".
no-store
Cache-Control: no-store — "אל תשמור בכלל"
לנתונים רגישים: בנק, מסמכים אישיים. הכי יקר — כל בקשה הולכת לשרת.
stale-while-revalidate
stale-while-revalidate=60 — "הגש מה-Cache, ועדכן ברקע"
המשתמש מקבל תשובה מיידית. Cache מתעדכן בשקט. Best of both worlds.
ETag — "האם השתנה מאז?"
🏷️ ETag כמו תאריך על מוצר:
Server מחזיר ETag: "abc123" — Hash של התוכן.
בבקשה הבאה Client שולח: If-None-Match: "abc123".
אם לא השתנה → Server מחזיר 304 Not Modified ללא Body. חסכנו 100% Bandwidth.
Rate Limiting — הגנה על השרת
למה?
ללא Rate Limiting — מישהו יכול לשלוח מיליון בקשות ולהפיל את השרת (DDoS).
429
כשעוברים את הגבול: 429 Too Many Requests + Retry-After: 30
אירוע אמיתי
💸 COSTLY — 2023
Header אחד שגוי גרם לחשבון ענן של 50,000 דולר בחודש
Startup ישראלית שלחה 100% מהRequests ל-Origin כי Cache-Control: no-store היה ברירת המחדל.

AWS Bill: $50K/חודש במקום $3K. שינוי ל-max-age=300 חסך 94% מהעלויות.

Header אחד. 5 דקות תיקון. 47,000 דולר חיסכון לחודש.
📖 מילון מושגים — פרק זה ▶
Cacheאחסון זמני של תוצאות כדי לא לחשב/לשאול שוב. יותר מהיר, פחות יקר.
Cache-ControlHTTP Header שמגדיר מדיניות Cache: כמה זמן, מה מותר, איפה.
max-ageכמה שניות ה-Cache תקף. max-age=300 = 5 דקות.
ETagHash של תוכן Response. מאפשר בדיקת "האם השתנה?" — 304 אם לא.
304 Not Modifiedתשובה ללא Body — "התוכן שיש לך עדיין תקף". חוסך Bandwidth.
CDNContent Delivery Network — שרתי Cache מפוזרים גיאוגרפית. Cloudflare, AWS CloudFront.
Rate Limitingהגבלת בקשות לזמן נתון. מגן מ-DDoS ושימוש לא הוגן.
429Too Many Requests — עברת את מגבלת ה-Rate Limit.
ARCH
API Design Patterns
API Gateway, BFF, Webhooks, gRPC, Circuit Breaker, OpenAPI
ARCH⏱ 75m▶
כשיש לך יותר מ-API אחד, מספר Clients, ומיקרו-שירותים — צריך ארכיטקטורה. הפרק הזה על Patterns שמופיעים שוב ושוב בחברות גדולות: API Gateway, BFF, Webhooks, Circuit Breaker.
API Gateway — שוער בכניסה
🏨 API Gateway כמו קונסיירז' במלון:
במקום שכל אורח ילך ישירות לשף/ניקיון/חשמלאי — פונה לקונסיירז' שמנתב לכולם.
API Gateway עושה: Auth (מי אתה?), Rate Limiting, Logging, ניתוב לשירות הנכון — במקום אחד.
BFF — API שנתפר לClient
בעיה
Web App צריך נתונים מלאים. Mobile App צריך גרסה רזה לחסוך Bandwidth. CLI צריך פורמט שונה. API אחד לא מתאים לכולם.
BFF
Backend for Frontend — API נפרד לכל Client. Web BFF, Mobile BFF, CLI BFF. כל אחד מחזיר בדיוק מה שה-Client צריך.
Webhooks — Push במקום Poll
🔔 Webhook כמו התראת Push:
Polling = כל דקה שואל "יש הזמנה חדשה?" — בזבוז גדול.
Webhook = Stripe שולח לך HTTP POST ברגע שיש תשלום.

אתה נותן ל-Stripe את ה-URL שלך. הם "דוחפים" אליך עדכון כשקורה משהו.
Circuit Breaker — מניעת קריסה מדבקת
⚡ Circuit Breaker כמו ממסר חשמלי:
כשיש קצר — ממסר מתנתק כדי שהאש לא תתפשט.
כשה-Payment Service קורס — Circuit Breaker מפסיק לשלוח בקשות אליו (במקום לחכות Timeout כל פעם) ומחזיר שגיאה מיידית. אחרי Timeout — מנסה שוב בהדרגה.
אירוע אמיתי
🔴 CASCADE — 2022 / Meta
איך ניסיון לתקן עצמו גרם לקריסה של Instagram
Backend הואט → Clients שלחו Retries ללא Exponential Backoff. כל Retry → יותר Load → יותר Retries → Thundering Herd.

Circuit Breaker + Exponential Backoff היו מונעים: "נכשל פעם 1 → חכה 1 שנ'. פעם 2 → חכה 2 שנ'. פעם 3 → חכה 4 שנ'."

Jitter (אקראיות קטנה) מונע שכולם ינסו שוב בדיוק באותו רגע.
📖 מילון מושגים — פרק זה ▶
API Gatewayשרת שיושב לפני כל ה-APIs. מטפל ב-Auth, Rate Limiting, Routing. Kong, AWS API GW.
BFFBackend for Frontend — API ייחודי לכל סוג Client: Web, Mobile, CLI.
WebhookServer שולח HTTP POST ל-URL שלך כשקורה אירוע. Push במקום Poll.
Circuit BreakerPattern שמפסיק לשלוח לService תקוע. מונע Cascading Failure.
Exponential Backoffהמתנה גוברת בין Retries: 1s → 2s → 4s → 8s. מונע הצפת שרת.
Jitterאקראיות קטנה ב-Backoff כדי שכולם לא ינסו שוב בדיוק אותו רגע.
Thundering Herdגל של בקשות שמגיעות בבת אחת — בד"כ Retries מסונכרנים. מרסק שרתים.
gRPCGoogle RPC — Binary protocol מהיר. Protocol Buffers במקום JSON. מצוין ל-Microservices.

Testing, Docs & Load Testing
Postman, Contract Testing, k6, OpenAPI, Chaos Engineering

API שעובד אבל לא מתועד — כאילו לא קיים. API שמתועד אבל לא נבדק — זמן פצצה. הפרק הזה על שלושה דברים שמבדילים API מקצועי מהפקר: תיעוד, בדיקות, ו-Load Testing.
OpenAPI — Spec שיוצר הכל
📐 OpenAPI כמו תוכניות אדריכל:
תוכנית אחת → ממנה ניתן לבנות בניין, לייצר IKEA instructions, ולקבל אישור עירייה.
Spec OpenAPI אחד → ממנו ניתן לייצר: Docs אינטראקטיביות, SDK בכל שפה, Tests אוטומטיים.
YAML

Swagger

SDK

Contract Testing — מניעת Breaking Changes
📋 Contract Testing כמו חוזה:
ה-Frontend מצפה לקבל {"user": {"name": "Dana"}}.
ה-Backend משנה ל-{"user": {"fullName": "Dana"}} — Frontend נשבר.
Contract Testing (Pact) בודק שכל שינוי ב-API לא שובר Client קיים, לפני שמפרסמים.
Load Testing — כמה עומד ה-API?
k6
כלי Load Testing: מדמה אלפי VUs (Virtual Users) ובודק מה קורה ל-p95 Latency תחת עומס.
p95
Percentile 95: "95% מהבקשות היו מהירות מ-X ms". הממוצע מסתיר בעיות — p95 חושף אותן.
Threshold
הגדר: "אם p95 > 500ms — Test נכשל". כך Load Test הופך לחלק אוטומטי מה-CI.
