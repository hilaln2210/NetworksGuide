מה זה API בכלל? — מבוא מאפס
לגמרי מהתחלה · ללא ידע מוקדם · אנלוגיות מהחיים
התחלה ⏱ 25m ▶
לפני שנדבר על HTTP, REST, JSON ו-Endpoints — צריך לענות על שאלה אחת: מה בעצם קורה כשאפליקציה "מדברת" עם שרת?

אם אתה פותח Waze, מזמין ב-Wolt, או מתחבר עם Google לאפליקציה אחרת — כל אחד מהרגעים האלה מערב API. הפרק הזה מסביר מה זה מהתחלה, ללא ידע מוקדם.
מה זה API?
🍽️ האנלוגיה של המסעדה:
דמיין מסעדה. אתה (הלקוח) יושב בחדר האוכל. המטבח הוא השרת. אתה לא נכנס למטבח — אתה קורא למלצר.

ה-API הוא המלצר.
אתה מזמין (Request), המלצר הולך למטבח, ומביא לך את המנה (Response). אתה לא יודע ולא צריך לדעת איך הבישול עובד.
01
API = Application Programming Interface = ממשק שמאפשר לשתי תוכנות לדבר אחת עם השנייה. זה לא קוד מיוחד — זו הסכמה: "שלח לי בקשה בפורמט הזה, ואני אחזיר לך תשובה בפורמט הזה."
02
דוגמה מהחיים: כשאתה לוחץ "Login with Google" באפליקציה — האפליקציה שולחת בקשה ל-API של Google, Google בודקת מי אתה, ומחזירה: "זה יוסי, המייל שלו הוא [email protected]". האפליקציה קיבלה מידע בלי לדעת את הסיסמה שלך.
03
למה זה חשוב? כמעט כל אפליקציה מודרנית בנויה מ-APIs — Waze קונה API של מפות מגוגל, Wolt מחבר בין API של מסעדות, API של תשלומים, API של מיקום. ה-APIs הם "הצינורות" שמחברים הכל.
Client ו-Server — מי מדבר עם מי?
📞 אנלוגיית הטלפון:
Client = מי שמתקשר (שואל שאלות, שולח בקשות)
Server = מי שעונה (מחזיק מידע, מעבד בקשות)

הדפדפן שלך הוא Client. כשאתה פותח youtube.com — הדפדפן מתקשר לשרתי YouTube ומבקש: "תן לי את הדף הראשי". השרת עונה עם HTML, תמונות, ווידאו.
💡 נקודה חשובה: Client ו-Server יכולים להיות כל דבר. אפליקציית מובייל היא Client. דפדפן הוא Client. אפילו שרת אחד יכול להיות Client של שרת אחר — כשהם מדברים ביניהם.
HTTP — שפת התקשורת של האינטרנט
אם API הוא המלצר, אז HTTP הוא השפה שבה הם מדברים. כשהדפדפן שלך מבקש דף — הוא שולח הודעה בפורמט HTTP. השרת עונה ב-HTTP.
שאל
Request (בקשה) — Client שולח לשרת:
GET /api/users/42 — "תן לי את פרטי המשתמש מספר 42"
POST /api/orders — "צור הזמנה חדשה עם הנתונים הבאים"
DELETE /api/posts/7 — "מחק את הפוסט מספר 7"
קבל
Response (תשובה) — Server מחזיר:
200 OK — הצליח, הנה הנתונים
404 Not Found — לא מצאתי את מה שביקשת
500 Server Error — משהו השתבש אצלי
🔢 Status Codes — ספרות שמספרות את הסיפור:
200-299 = ✅ הצליח
300-399 = ↗️ הועבר לכתובת אחרת
400-499 = ❌ טעית בבקשה (הלקוח אשם)
500-599 = 💥 שגיאת שרת (השרת אשם)

בדיוק כמו ציונים בבית ספר — המספר מספר לך בלי לקרוא הרבה אם זה עבר או נכשל.
JSON — פורמט הנתונים
שלחנו בקשה, קיבלנו תשובה — אבל באיזה פורמט מגיעים הנתונים? כמעט תמיד ב-JSONJSONJavaScript Object Notation — פורמט טקסטואלי פשוט לייצוג נתונים. קריא לבני אדם ולמחשבים..
מה
JSON נראה כך:
{ "name": "Dana", "age": 28, "city": "Tel Aviv" }

זה פשוט טקסט מאורגן. מפתח (name) ואחריו ערך ("Dana"). כל API שאתה מדבר איתו יחזיר לך JSON כזה עם הנתונים שביקשת.
למה
למה JSON ולא משהו אחר?
כי כל שפת תכנות יכולה לקרוא אותו. Python, JavaScript, Java — כולם "מדברים" JSON. זה הפך לשפת האם של ה-Web.
הכל יחד — מה קורה כשאתה פותח Wolt?
1
האפליקציה (Client) שולחת בקשת HTTP לשרת של Wolt: GET /api/restaurants?city=tlv
2
שרת Wolt (Server) מקבל את הבקשה, שולף מסד הנתונים את המסעדות בתל אביב
3
השרת מחזיר JSON: [{"name": "בורגר סאלים", "rating": 4.8}, ...]
4
האפליקציה קוראת את ה-JSON ומציגה את המסעדות על המסך
🎯 הרגע שבו הכל מתחבר: כל הפרקים הבאים מרחיבים על שלב אחד מהתהליך הזה. HTTP → REST → Auth → WebSockets. אבל הבסיס הוא תמיד: Client שואל, Server עונה, JSON מחזיר נתונים.

HTTP — הפרוטוקול שמריץ את האינטרנט
Request, Response, Headers, Methods, Status Codes, HTTP/2, HTTP/3
CORE⏱ 50m▶
כל פעם שאתה פותח אתר, שולח הודעה, או מחפש ב-Google — הדפדפן שלך שולח בקשת HTTP לשרת מרוחק. HTTP הוא השפה הבסיסית שבה כל הרשת מדברת. לפני שנבין API — חייבים להבין את השפה עצמה.
📬 HTTP כמו מערכת דואר מאוד מסודרת:
כל בקשה היא מעטפה עם כתובת מדויקת, סוג מסר (מכתב? חבילה? טופס?), ותוכן. כל תשובה מגיעה עם קוד שאומר: "נמסר ✅", "לא נמצא ❌", "תקלה בדואר 💥". אתה לא יוצר קשר ישיר עם הנמען — הכל עובר דרך המערכת.
Request — מה שולחים
Method
מה אתה רוצה לעשות?
GET — תן לי מידע (קריאה בלבד)
POST — צור משהו חדש
PUT — החלף משהו קיים במלואו
PATCH — עדכן חלק ממשהו קיים
DELETE — מחק משהו
URL
לאן? הכתובת המדויקת: https://api.example.com/users/42
https:// = פרוטוקול מאובטח
api.example.com = הדומיין / השרת
/users/42 = ה-Endpoint הספציפי
Headers
מטא-דאטה על הבקשה:
Content-Type: application/json — "שולח לך JSON"
Authorization: Bearer eyJ... — "הנה הטוקן שלי"
Accept: application/json — "אני מצפה לקבל JSON"
Body
התוכן עצמו (רק ב-POST/PUT/PATCH):
{"name": "Dana", "email": "[email protected]"}
ב-GET אין Body — כל הנתונים ב-URL עצמו.
Status Codes — מה הייתה התשובה
🚦 5 קטגוריות, מספר אחד מספר את הכל:
1xx — מידע בלבד (נדיר)
2xx — ✅ הצליח (200 OK, 201 Created, 204 No Content)
3xx — ↗️ Redirect (301 Moved, 304 Not Modified)
4xx — ❌ הלקוח טעה (400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests)
5xx — 💥 השרת שגה (500 Internal Error, 503 Service Unavailable)
HTTP/1.1 → HTTP/2 → HTTP/3
1.1
HTTP/1.1 (1997) — בקשה אחת בכל פעם על כל חיבור. כמו תור לקופה — כולם מחכים לאחד.
2
HTTP/2 (2015) — כמה בקשות במקביל על אותו חיבור (Multiplexing). Header Compression. פי 2-3 מהיר.
3
HTTP/3 (2022) — רץ על UDP במקום TCP. אם Packet אבד — רק ה-Stream שלו מחכה, לא כולם. מצוין ב-Mobile.
💡 Statelessness — עיקרון מרכזי: כל בקשת HTTP עצמאית לחלוטין. השרת לא זוכר שביקרת לפני. לכן צריך לשלוח Token בכל בקשה — השרת לא "זוכר" שהתחברת.

REST API — תכנון שנעשה נכון
Resources, Naming, Versioning, Pagination, Idempotency, RFC 7807
REST⏱ 60m▶
HTTP הוא פרוטוקול — שפה. REST הוא סגנון ארכיטקטורה — כללים איך לתכנן את ה-API שלך על גבי HTTP. רוב ה-APIs שתפגוש (GitHub, Stripe, Twitter) הם REST APIs. אבל REST API גרוע הוא אחד הדברים הכי מתסכלים שיש.
🗂️ REST כמו ספרייה מסודרת:
בספרייה יש Resources (ספרים, מחברים, קטגוריות) — לכל אחד כתובת ברורה. אתה לא אומר "תן לי את ספרי המדע של ג'ון" — אתה הולך ל /authors/john/books?genre=science. ה-URL מתאר מה אתה מחפש, ה-Method מתאר מה אתה עושה.
כלל 1 — Resource Naming
❌
שמות שגויים (RPC-style):
/getUser?id=42
/createNewOrder
/deleteUserAccount/5
ה-URL מתאר פעולה — זה לא REST.
✅
שמות נכונים (Resource-style):
GET /users/42 — קרא משתמש
POST /orders — צור הזמנה
DELETE /users/5 — מחק משתמש
ה-URL מתאר ישות, ה-Method מתאר פעולה.
כלל 2 — Idempotency
🔁 Idempotency = "אפשר לשלוח פעמיים בלי נזק":
דמיין שלחצת "שלם" ב-Wolt והרשת נפלה — לא ידוע אם ההזמנה עברה. האם בטוח ללחוץ שוב?

GET, PUT, DELETE — כן. שלח כמה פעמים שרוצה — אותה תוצאה.
POST — לא! שלח פעמיים = שתי הזמנות. לכן Stripe משתמש ב-Idempotency Keys.
כלל 3 — Versioning
למה?
API שמשתמשים בו לא יכול "פשוט להשתנות" — 10,000 אפליקציות יישברו. צריך גרסאות.
איך?
URL (הנפוץ ביותר): /api/v1/users → /api/v2/users
Header: Accept: application/vnd.api+json;version=2
Query Param: /api/users?version=2
כלל 4 — Pagination
בעיה
מה קורה אם יש מיליון משתמשים? לא מחזירים הכל — מחזירים בדפים.
OFFSET
GET /users?limit=20&offset=40 — "תן לי 20 מהשורה ה-40"
בעיה: אם נוסף user בזמן הגלילה — תפסיד שורה. כבד ב-DB על טבלאות גדולות.
Cursor
GET /users?limit=20&after=user_abc123 — "תן לי 20 אחרי המשתמש הזה"
עדיף: מהיר יותר, לא מפספס נתונים. זה מה-Twitter/Instagram משתמשים בו.
💡 RFC 7807 — Error Response סטנדרטי: במקום להחזיר {"error": "bad"} — החזר: {"type": "...", "title": "Validation Failed", "status": 400, "detail": "email is required"}. כל Client יכול לטפל בשגיאות בצורה אחידה.

GraphQL — בקש רק מה שאתה צריך
Schema, Queries, Mutations, Subscriptions, N+1, DataLoader
GRAPHQL⏱ 70m▶
REST עם כל כלליו הוא מצוין — אבל יש בעיה: אתה מקבל מה שהשרת מחליט לתת לך. נניח שאתה צריך רק שם ותמונה של User — REST מחזיר 20 שדות. GraphQL פותר זאת: הClient מחליט בדיוק מה לקבל.
🍕 GraphQL כמו פיצה Build-Your-Own:
ב-REST: "הגבינה עם הפטריות" — תקבל את מה שהמסעדה מחליטה שנמצא בפיצה הזו.
ב-GraphQL: "אני רוצה בצק דק, עגבנייה, גבינה, ורק זיתים" — אתה מזמין בדיוק מה שצריך.

Facebook המציאה אותו ב-2012 כי ה-Mobile App שלהם היה איטי — מורידים יותר מדי נתונים.
Over-fetching ו-Under-fetching
❌
Over-fetching ב-REST: צריך רק name ו-avatar של User — מקבל 15 שדות כולל address, preferences, metadata...
❌
Under-fetching ב-REST: רוצה User + Orders שלו + פרטי Product בכל Order — צריך 3 בקשות נפרדות.
✅
GraphQL: בקשה אחת, מקבל בדיוק מה שביקשת:
query {"{"} user(id:"42") {"{"} name avatar orders {"{"} total {"}"} {"}"} {"}"}
N+1 Problem — הבעיה הכי נפוצה ב-GraphQL
⚠️ מה זה N+1 ולמה זה מסוכן:
בקשת 10 Users עם ה-Orders שלהם = 1 שאילתת DB לUsers + 10 שאילתות לOrders = 11 שאילתות.
בקשת 100 Users = 101 שאילתות. 1000 = 1001.

הפתרון: DataLoader — מרכז את כל הבקשות ל-Orders וב-Batch אחד: 1+1=2 שאילתות בלבד.
מתי GraphQL ומתי REST?
REST
עדיף כשמה: Public API פשוט, Caching חשוב, הצוות מכיר REST, CRUD בסיסי
GraphQL
עדיף כשמה: כמה Clients עם צרכים שונים (Web/Mobile), נתונים מורכבים ומקוננים, הצוות מנוסה

🏢 כניסה לבניין משרדים:

JWT — הToken הנפוץ ביותר
מה?
JWT = JSON Web Token — טוקן מקודד עם 3 חלקים מופרדים בנקודה:
eyJhbGci.eyJzdWIiOiI0Mn0.SflKxwRJSMeK
מבנה
Header — אלגוריתם ההצפנה (HS256/RS256)
Payload — גלוי לכולם! מכיל user_id, roles, תפוגה
Signature — חתימה שמאמתת שה-Token לא שונה
⚠️
Payload גלוי לכולם — Base64 בלבד, לא הצפנה. אל תשים סיסמאות ב-Payload!

3

PERF⏱ 70m▶
הAPI שלך עובד — אבל כמה מהר? ו-כמה עולה? Caching הוא הדרך לענות על שאלות שכבר נשאלו בלי לעבוד מחדש. Cache-Control הוא ה-Header שקובע מה, כמה, ואיך.
🗂️ Cache כמו פתק הליד:
במקום לשאול כל פעם "מה מספר הטלפון של עמיר?" — כתבת אותו על פתק. בפעם הבאה — קרא מהפתק.
אבל אם עמיר שינה מספר ואתה לא יודע — הפתק מוטעה. לכן צריך להגדיר: "הפתק הזה תקף X דקות".
Cache-Control — Header שמגדיר הכל
max-age
Cache-Control: max-age=300 — "שמור 5 דקות, אל תשאל שוב"
המשתמש לא מחכה, Cache עונה מיידית. אחרי 5 דקות — בקשה חדשה.
no-cache
Cache-Control: no-cache — "שמור, אבל בדוק תמיד אם עדכני"
לא אומר "אל תשמור" — אומר "אמת לפני שימוש".
no-store
Cache-Control: no-store — "אל תשמור בכלל"
לנתונים רגישים: בנק, מסמכים אישיים. הכי יקר — כל בקשה הולכת לשרת.
stale-while-revalidate
stale-while-revalidate=60 — "הגש מה-Cache, ועדכן ברקע"
המשתמש מקבל תשובה מיידית. Cache מתעדכן בשקט. Best of both worlds.
ETag — "האם השתנה מאז?"
🏷️ ETag כמו תאריך על מוצר:
Server מחזיר ETag: "abc123" — Hash של התוכן.
בבקשה הבאה Client שולח: If-None-Match: "abc123".
אם לא השתנה → Server מחזיר 304 Not Modified ללא Body. חסכנו 100% Bandwidth.

למה?
ללא Rate Limiting — מישהו יכול לשלוח מיליון בקשות ולהפיל את השרת (DDoS).
429
כשעוברים את הגבול: 429 Too Many Requests + Retry-After: 30

What is an API? — From Zero
Complete beginner · No prior knowledge · Real-life analogies
Beginner ⏱ 25m ▶
Before we talk about HTTP, REST, JSON and Endpoints — we need to answer one question: What actually happens when an app "talks" to a server?

When you open Waze, order food, or sign in with Google — every one of those moments involves an API. This chapter explains what that means from scratch, no prior knowledge needed.
What is an API?
🍽️ The Restaurant Analogy:
Imagine a restaurant. You (the customer) are in the dining room. The kitchen is the server. You don't enter the kitchen — you call the waiter.

The API is the waiter.
You order (Request), the waiter goes to the kitchen, and brings back your dish (Response). You don't know — and don't need to know — how the cooking works.
01
API = Application Programming Interface = an interface that lets two pieces of software talk to each other. It's not special code — it's an agreement: "send me a request in this format and I'll return a response in this format."
02
Real-world example: When you click "Login with Google" — the app sends a request to Google's API, Google checks who you are, and returns: "This is Dana, their email is [email protected]". The app got the info without ever knowing your password.
03
Why does this matter? Almost every modern app is built from APIs — maps apps buy a mapping API, food delivery apps connect a restaurant API, a payment API, and a location API. APIs are the "pipes" that connect everything.
Client & Server — Who Talks to Whom?
📞 The Phone Call Analogy:
Client = the caller (asks questions, sends requests)

HTTP — The Protocol That Runs the Internet
Request, Response, Headers, Methods, Status Codes, HTTP/2, HTTP/3
CORE⏱ 50m▶
Every time you open a website, send a message, or search Google — your browser sends an HTTP request to a remote server. HTTP is the fundamental language the whole web speaks. Before we understand APIs — we need to understand the language itself.
📬 HTTP Like a Very Organized Postal System:
Every request is an envelope with a precise address, message type (letter? package? form?), and content. Every reply comes with a code: "Delivered ✅", "Not Found ❌", "Postal failure 💥". You don't contact the recipient directly — everything goes through the system.
Request — What You Send
Method
What do you want to do?
GET — give me information (read only)
POST — create something new
PUT — fully replace something existing
PATCH — partially update something existing
DELETE — delete something
URL
Where to? The exact address: https://api.example.com/users/42
https:// = secure protocol
api.example.com = the domain / server
/users/42 = the specific Endpoint
Headers
Metadata about the request:
Content-Type: application/json — "sending you JSON"
Authorization: Bearer eyJ... — "here's my token"
Accept: application/json — "I expect to receive JSON"
Body
The content itself (only in POST/PUT/PATCH):
{"name": "Dana", "email": "[email protected]"}
GET has no Body — all data is in the URL itself.
Status Codes — What Was the Answer
🚦 5 categories, one number tells the whole story:
1xx — Informational (rare)
2xx — ✅ Success (200 OK, 201 Created, 204 No Content)
3xx — ↗️ Redirect (301 Moved, 304 Not Modified)
4xx — ❌ Client error (400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests)
5xx — 💥 Server error (500 Internal Error, 503 Service Unavailable)
HTTP/1.1 → HTTP/2 → HTTP/3
1.1
HTTP/1.1 (1997) — one request at a time per connection. Like a queue at checkout — everyone waits for the one in front.
2
HTTP/2 (2015) — multiple requests in parallel on the same connection (Multiplexing). Header Compression. 2-3x faster.
3
HTTP/3 (2022) — runs on UDP instead of TCP. If a Packet is lost — only its Stream waits. Great on Mobile.
💡 Statelessness — a core principle: every HTTP request is completely independent. The server doesn't remember your previous visit. That's why you send a Token with every request — the server doesn't "remember" that you logged in.
Real Incident
🔴 BREACH — 2020 / Zoom
How Zoom Calls Between Americans Were Routed Through China
In early 2020, Citizen Lab discovered that Zoom's encryption keys were being routed through servers in China — even when both parties were in the US.

The reason: a routing mechanism that found the nearest available server — without checking geographic location. A server in China responded faster? The key went through there.

The fix: adding Geo-Fencing — a rule saying "keys for American calls stay on American servers only." Cost: $1 billion in settlements.

REST API — Design Done Right
Resources, Naming, Versioning, Pagination, Idempotency, RFC 7807
REST⏱ 60m▶
HTTP is a protocol — a language. REST is an architectural style — rules for designing your API on top of HTTP. Most APIs you'll encounter (GitHub, Stripe, Twitter) are REST APIs. But a poorly designed REST API is one of the most frustrating things.
🗂️ REST Like a Well-Organized Library:
A library has Resources (books, authors, categories) — each with a clear address. You don't say "give me John's science books" — you go to /authors/john/books?genre=science. The URL describes what you're looking for, the Method describes what you're doing.
Rule 1 — Resource Naming
❌
Wrong naming (RPC-style):
/getUser?id=42
/createNewOrder
/deleteUserAccount/5
The URL describes an action — this is not REST.
✅
Correct naming (Resource-style):
GET /users/42 — read user
POST /orders — create order
DELETE /users/5 — delete user
The URL describes an entity, the Method describes an action.
Rule 2 — Idempotency
🔁 Idempotency = "safe to send twice without side effects":
Imagine you clicked "Pay" in a food app and the network dropped — unknown if the order went through. Is it safe to click again?

GET, PUT, DELETE — yes. Send as many times as you want — same result.
POST — no! Send twice = two orders. That's why Stripe uses Idempotency Keys.
Rule 3 — Versioning
Why?
An API that people use can't "just change" — 10,000 apps would break. You need versions.
How?
URL (most common): /api/v1/users → /api/v2/users
Header: Accept: application/vnd.api+json;version=2
Query Param: /api/users?version=2
Rule 4 — Pagination
Problem
What if there are a million users? Don't return everything — return pages.
OFFSET
GET /users?limit=20&offset=40 — "give me 20 starting at row 40"
Problem: if a user is added during scrolling — you'll miss a row. Heavy on the DB for large tables.
Cursor
GET /users?limit=20&after=user_abc123 — "give me 20 after this user"
Better: faster, no missed data. This is what Twitter/Instagram use.
💡 RFC 7807 — Standard Error Response: Instead of returning {"error": "bad"} — return: {"type": "...", "title": "Validation Failed", "status": 400, "detail": "email is required"}. Every Client can handle errors in a consistent way.
Real Incident
🔴 BREAK — 2018 / Twitter
When Twitter Killed Millions of Apps in One Day
In 2013, Twitter announced the deprecation of API v1 for 2018. 5 years seemed like a lot — but 2018 arrived and many apps crashed.

The problem: no Sunset Header telling every Client "I'm closing on date X." Developers who didn't follow the blog learned the news when their service went down.

The lesson that became a standard: Sunset: Sat, 01 Jan 2018 00:00:00 GMT in every Response + 6 months of parallel support.

GraphQL — Request Only What You Need
Schema, Queries, Mutations, Subscriptions, N+1, DataLoader
GRAPHQL⏱ 70m▶
REST with all its rules is great — but there's a problem: you get what the server decides to give you. Say you only need a user's name and avatar — REST returns 20 fields. GraphQL solves this: the Client decides exactly what to receive.
🍕 GraphQL Like Build-Your-Own Pizza:
In REST: "the cheese with mushrooms" — you get whatever the restaurant decides is in that pizza.
In GraphQL: "I want thin dough, tomato, cheese, and only olives" — you order exactly what you need.

Facebook invented it in 2012 because their Mobile App was slow — downloading too much data.
Over-fetching & Under-fetching
❌
Over-fetching in REST: you only need name and avatar — you receive 15 fields including address, preferences, metadata...
❌
Under-fetching in REST: want User + their Orders + Product details in each Order — needs 3 separate requests.
✅
GraphQL: one request, get exactly what you asked for:
query {"{"} user(id:"42") {"{"} name avatar orders {"{"} total {"}"} {"}"} {"}"}
N+1 Problem — The Most Common GraphQL Pitfall
⚠️ What is N+1 and why is it dangerous:
Requesting 10 Users with their Orders = 1 DB query for Users + 10 queries for Orders = 11 queries.
Requesting 100 Users = 101 queries. 1000 = 1001.

The solution: DataLoader — groups all Order requests into one Batch: 1+1=2 queries total.
When GraphQL, When REST?
REST
Prefer when: simple public API, Caching matters, team knows REST, basic CRUD
GraphQL
Prefer when: multiple Clients with different needs (Web/Mobile), complex nested data, experienced team
Real Incident
🔴 OVERLOAD — 2019 / Shopify
The GraphQL Query That Took Down a Database
Without query depth limits, a developer wrote: user → friends → friends → orders → items → reviews (depth 7).

100 such requests = millions of DB queries = crash.

Shopify's solution: Depth Limit=5, Complexity Score per query, Rate Limiting by query cost.

GraphQLQuery language for APIs. The Client defines exactly what to receive. Created by Facebook in 2012.
Over-fetchingReceiving more data than needed. Wastes Bandwidth and is slower.
Under-fetchingNeeds multiple requests to gather all the required data.
SchemaDefinition of all types, fields, and possible queries in a GraphQL API.
QueryRead request in GraphQL — like GET in REST.
MutationWrite request in GraphQL — like POST/PUT/DELETE in REST.

Authentication & Authorization
JWT, OAuth 2.0, OpenID Connect, PKCE, API Keys, CSRF
SECURITY⏱ 80m▶
So far we've learned to send requests and receive data — but who is authorized to ask? Authentication and Authorization sound similar but are completely different. Without them, every API is open to everyone.
🏢 Entering an Office Building:
Authentication = who are you? (ID card, fingerprint, password)
Authorization = what are you allowed to do? (only floor 3, not the server room)

You can be authenticated (Authentication ✅) and still blocked (Authorization ❌).
JWT — The Most Common Token
What?
JWT = JSON Web Token — an encoded token with 3 parts separated by dots:
eyJhbGci.eyJzdWIiOiI0Mn0.SflKxwRJSMeK
Structure
Header — the signing algorithm (HS256/RS256)
Payload — visible to everyone! contains user_id, roles, expiry
Signature — signature that verifies the Token wasn't modified
⚠️
Payload is visible to everyone — Base64 only, not encryption. Never put passwords in the Payload!
OAuth 2.0 — Login with Google/GitHub
🔑 OAuth 2.0 Like a Valet Key:
Instead of giving someone your house key — you give them a limited valet key that only opens the front door for one hour.

When you click "Login with Google": the app gets a limited Token that lets it read your Email — not your Google password.
1
User clicks "Login with Google" → app redirects to Google
2
User authorizes Google to share their Email with the app
3
Google returns an Authorization Code to the app
4
App exchanges Code for Access Token → uses it to read Email
Real Incident
🔴 BREACH — 2022 / Optus
9.8 Million Australians Affected by API With No Password
An Optus Endpoint exposed data with zero Authentication. A simple script enumerated IDs and pulled 9.8M records.

No Token, no API Key, nothing. Name, address, passport number — visible to anyone who knew the URL.

Cost: $1.5 billion. Lesson: every Endpoint that exposes data must require Authentication.

WebSockets & Real-Time
WebSocket, SSE, Long Polling, WebRTC, Socket.io, Scaling
REAL-TIME⏱ 60m▶
HTTP is Request-Response: you ask, the server answers, the connection closes. But what if the server needs to push you an update — a new WhatsApp message, a stock price changing, a live location? It can't wait for you to ask. That's where WebSockets and SSE come in.
📞 HTTP vs WebSocket:
HTTP = SMS. Send a message, get a reply, conversation ends. Every question needs a new SMS.
WebSocket = a phone call. Open connection, both sides talk continuously, until someone hangs up.

WebSocket starts as HTTP, then "upgrades" to a persistent two-way connection.
WebSocket Handshake — How It Works
1
Client sends a regular HTTP request with a special Header: Upgrade: websocket
2
Server confirms: returns 101 Switching Protocols
3
The TCP connection stays open. Now both sides can send Messages whenever — no new request needed.
SSE — When You Only Need One Direction
📻 SSE Like a Radio Broadcast:
Server sends, Client receives — one-directional. Client can't send back on the same Channel.
When is SSE better than WebSocket? Notifications, Live Feed, Dashboard Updates — when you only need to receive updates, not send.
Advantage: simpler, built-in Auto-Reconnect, works over HTTP/2.
Real Incident
🔴 OUTAGE — 2020 / Discord
COVID Broke Discord in a Week
COVID brought growth from 2M to 5M Concurrent WebSocket Connections in days. Years of planned growth happened in a week.

WebSocket differs from HTTP: every user holds an open, persistent connection. 5M open connections = a completely different type of load.

The solution: Redis Pub/Sub — every WebSocket server listens for events through Redis, so a message from Server A reaches a User connected to Server B.

WebSocketProtocol for persistent two-way connection. Starts as HTTP and upgrades to Full-Duplex.
SSEServer-Sent Events — one-way push from Server to Client. Simple, Auto-Reconnect.
Long PollingClient sends a request, Server "holds" it until there's news. Less efficient than WebSocket.
HandshakeProcess of opening a WebSocket connection — HTTP Upgrade Request + 101 Response.
Full-DuplexBoth sides send and receive simultaneously. A phone call is Full-Duplex, SMS is not.

Performance & Caching
CDN, Cache-Control, ETags, Rate Limiting, Compression, Connection Pool
PERF⏱ 70m▶
Your API works — but how fast? and how much does it cost? Caching is the way to answer questions already asked without doing the work again. Cache-Control is the Header that defines what, how long, and how.
🗂️ Cache Like a Sticky Note:
Instead of asking every time "what's Amir's phone number?" — you wrote it on a sticky note. Next time — read the note.
But if Amir changed their number and you don't know — the note is wrong. So you need to define: "this note is valid for X minutes".
Cache-Control — The Header That Defines Everything
max-age
Cache-Control: max-age=300 — "keep for 5 minutes, don't ask again"
User doesn't wait, Cache answers immediately. After 5 minutes — new request.
no-cache
Cache-Control: no-cache — "keep, but always verify if current"
Doesn't mean "don't save" — means "validate before use".
no-store
Cache-Control: no-store — "don't store at all"
For sensitive data: banking, personal documents. Most expensive — every request goes to the server.
stale-while-revalidate
stale-while-revalidate=60 — "serve from Cache, update in background"
User gets an immediate response. Cache updates silently. Best of both worlds.
ETag — "Has It Changed?"
🏷️ ETag Like a Product Timestamp:
Server returns ETag: "abc123" — a Hash of the content.
On the next request Client sends: If-None-Match: "abc123".
If unchanged → Server returns 304 Not Modified with no Body. We saved 100% Bandwidth.
Rate Limiting — Protecting the Server
Why?
Without Rate Limiting — someone could send a million requests and take down the server (DDoS).
429
When the limit is exceeded: 429 Too Many Requests + Retry-After: 30
Real Incident
💸 COSTLY — 2023
One Wrong Header Cost $50,000/Month in Cloud Bills
An Israeli startup sent 100% of Requests to Origin because Cache-Control: no-store was the default.

AWS Bill: $50K/month instead of $3K. Changing to max-age=300 saved 94% of costs.

One Header. 5 minutes to fix. $47,000 saved per month.

API Design Patterns
API Gateway, BFF, Webhooks, gRPC, Circuit Breaker, OpenAPI
ARCH⏱ 75m▶
When you have more than one API, multiple Clients, and microservices — you need architecture. This chapter covers Patterns that appear again and again at large companies: API Gateway, BFF, Webhooks, Circuit Breaker.
API Gateway — The Gatekeeper
🏨 API Gateway Like a Hotel Concierge:
Instead of every guest going directly to the chef/cleaning/electrician — they go to the concierge who routes to everyone.
API Gateway does: Auth (who are you?), Rate Limiting, Logging, routing to the right service — in one place.
BFF — API Tailored to Each Client
Problem
Web App needs full data. Mobile App needs a lean version to save Bandwidth. CLI needs a different format. One API doesn't fit all.
BFF
Backend for Frontend — a separate API for each Client. Web BFF, Mobile BFF, CLI BFF. Each returns exactly what that Client needs.
Webhooks — Push Instead of Poll
🔔 Webhook Like a Push Notification:
Polling = every minute asking "is there a new order?" — huge waste.
Webhook = Stripe sends you an HTTP POST the moment there's a payment.

You give Stripe your URL. They "push" you an update when something happens.
Circuit Breaker — Preventing Cascading Failure
⚡ Circuit Breaker Like an Electrical Fuse:
When there's a short circuit — the fuse trips so the fire doesn't spread.
When the Payment Service crashes — Circuit Breaker stops sending requests to it (instead of waiting for Timeout every time) and returns an error immediately. After a Timeout — tries again gradually.
Real Incident
🔴 CASCADE — 2022 / Meta
How Auto-Recovery Caused Instagram to Crash
Backend slowed → Clients sent Retries without Exponential Backoff. Every Retry → more Load → more Retries → Thundering Herd.

Circuit Breaker + Exponential Backoff would have prevented it: "fail 1 → wait 1s. fail 2 → wait 2s. fail 3 → wait 4s."

Jitter (small randomness) prevents everyone from retrying at exactly the same moment.

The Frontend expects to receive {"user": {"name": "Dana"}}.

function completeStep(labN, stepId) {
  var ms = missionState[labN];
  if (!ms) return;
  if (ms.done.has(stepId)) return; // already done
  ms.done.add(stepId);

  // Tick the checkbox (both HE and EN)
  var el = document.getElementById(stepId);
  var elEn = document.getElementById(stepId + '_en');
  if (el) el.classList.add('completed');
  if (elEn) elEn.classList.add('completed');

  // Update badge
  var badge = document.getElementById('mission' + labN + '-badge');
  if (badge) {
    badge.querySelector('.mb-todo').innerHTML =
      '' + ms.done.size + ' / ' + ms.total + '' +
      '' + ms.done.size + ' / ' + ms.total + '';
    if (ms.done.size >= ms.total) {
      badge.classList.add('done');
      badge.querySelector('.mb-todo').innerHTML =
        '✅ הושלם!✅ Done!';
      // Show success
      var success = document.getElementById('mission' + labN + '-success');
      if (success) {
        success.style.display = 'flex';
        success.scrollIntoView({ behavior: 'smooth', block: 'nearest' });
      }
    }
  }
}
